## 1. Goal

Adapt the existing internal Clean Coders proprietary starter into an OSS-ready template under `templates/full-stack-reagent/`. The template tree is a runnable Clojure project at HEAD (literal `acme` tokens, no mustache) and is scaffolded by `c3kit-create` per the CLI sub-spec. Five optional features (`:content`, `:ssr`, `:csp`, `:markdownc`, `:auth`) and four DB backends (`:datomic-pro`, `:sqlite`, `:postgres`, `:memory`) are user-selectable at scaffold time.

---

## 2. Scope & Non-Goals

**In scope (v1):**

- Adapt proprietary starter file tree into `templates/full-stack-reagent/`.
- `c3kit-template.edn` manifest declaring tokens, secrets, features, DB options, next-steps, hook.
- Feature markers (`;; @c3kit/feature :<id> { … }`) inserted into shared files; per-feature whole-file deletes via manifest `:delete-when-off`.
- DB-selection markers (`;; @c3kit/db :<id> { … }`) in `config.clj` + `bucket.edn` + `deps.edn`.
- Optional `c3kit-template.bb` post-scaffold hook for DB-selection actions the marker mechanism cannot express (generating `bin/db` body, JDBC deps injection).
- OSS-flavored template `README.md` that ships in the scaffolded project, plus a separate template-dev `README.md` at `templates/full-stack-reagent/README.md` explaining how to work on the template itself.
- Template-level CI workflow that scaffolds the template via the CLI across a representative DB × feature matrix and runs `clj -M:test:spec` + `clj -M:test:cljs once` against each scaffold.

**Out of scope (v1):**

- New features beyond what already exists in the proprietary starter.
- Restructuring proprietary code (only adaptation: strip proprietary refs, add markers, drop `bin/setup`, replace `my.datomic.com` repo, drop AWS SDK deps).
- T5 (FE-only vanilla CLJS) — peer deliverable, separate sub-spec.
- Datomic Cloud / DynamoDB backends.
- AWS-based staging/production plumbing (`acme.email :ses`, AWS SDK deps). Documented as a "bring your own" step in the scaffolded README.
- `bin/setup` script (CLI owns rename).
- Homebrew/docker-compose distribution.
- Native Windows runners in CI (Phase 2+).

## 4. Open-Question Resolutions

These resolve the open questions tracked in the handoff and the roadmap §"Open questions for T1 sub-spec".

| Q | Resolution |
|---|---|
| Q1 — Datomic Pro install story | Modern `com.datomic/peer` is on Maven Central and free; the proprietary template's `my.datomic.com` mvn-repos entry is removed. Scaffolded README documents the Datomic Pro install: download the transactor zip from `https://www.datomic.com/get-datomic.html`, unzip to `~/.c3kit/datomic-pro/`, run via `bin/db`. `bin/db` is generated by the post-scaffold hook (see §7) when the user selects `:datomic-pro`. |
| Q2 — Bucket backends day one | `:datomic-pro`, `:sqlite` (JDBC), `:postgres` (JDBC), `:memory`. All four confirmed against c3kit/bucket. DynamoDB / Datomic Cloud deferred. |
| Q3 — `docker-compose.yml` | Not shipped. Datomic Pro runs as a JVM process (not docker); Postgres is the only backend that benefits and a single `brew install postgresql` or `apt install postgresql` is a one-liner. Adding docker as a prereq for OSS users is more friction than the saving. Instead, the hook generates a backend-specific `bin/db` script. |
| Q4 — Schema | When `:auth` on: `acme.schema/full` = `[bg-task] ++ user.schema/all`. When `:auth` off: `acme.schema/full` = `[bg-task]`. Marker block in `schema.cljc` controls the `user.schema/all` term. |
| Q5 — Seed | When `:auth` on: `dev/acme/seed.clj` ships 2 dev users (`road-runner@example.com`, `coyote@example.com`). When `:auth` off: `seed.clj` ships an empty `-main` no-op (file kept, marker block strips user fixtures). |
| Q6 — AWS SDKs | Dropped from `deps.edn` entirely. `acme.email/client-send-email :ses` defmethod removed; only `:to-log` ships. Scaffolded README has a "Production Email" section that documents adding `com.amazonaws/aws-java-sdk-ses` back and re-implementing the `:ses` defmethod. Saves ~50 MB of dep weight. |
| Q7 — JWT auth scope | The `:auth` feature controls all of: JWT middleware + anti-forgery + session middleware + `acme.user.*` namespaces (web/ajax/api/apple/google/core/corec/schema) + `acme.destination` + signin/signup/forgot-password/recover-password web routes + CLJS pages for forgot/recover/signin. |
| Q8 — Feature cut-points | All five features have clean cut-points. Marker work concentrates in 8 files: `deps.edn`, `src/clj/acme/{main,config,routes,http,layouts}.clj`, `src/cljs/acme/main.cljs`, `src/cljc/acme/schema.cljc`, `dev/acme/seed.clj`. Whole-file deletes preferred where possible. |
| Q9 — CI for the template | A dedicated GitHub Actions workflow at `.github/workflows/template-full-stack-reagent.yml` in the OSS repo (separate from CLI's own CI). Builds the CLI uberscript locally, then runs a matrix of `{db × feature-combo}` scaffolds (see §8). Phase 1 matrix is small (~8 combos) for speed. |
| Q10 — Per-env config | Keep dev/staging/production environments in `config.clj`. README notes that staging is optional. No marker gating (cheap to keep all three). |

### 6.3 `src/clj/acme/config.clj`

All four backends declare **distinct var names** at HEAD so the template tree compiles cleanly without any marker stripping. The `:bucket` value per env is set via line-level marker (one line wins per scaffold). HEAD's `:bucket` defaults to `memory-local` so the template at HEAD boots against the in-memory backend (success criterion §13).

```clojure
(ns acme.config
  (:require [c3kit.apron.app :as app]))

(def ^:private base
  {:analytics-code "console.log('google analytics would have loaded for this page');"
   :log-level      :info
   ;; @c3kit/feature :csp {
   :csp            {:enabled?       false
                    :enforce?       false
                    :report-handler nil
                    :policy         nil}
   ;; @c3kit/feature :csp }
   })

;; @c3kit/db :datomic-pro {
(def datomic-base
  {:impl                :datomic
   :migration-dir       "acme.migrations"
   :migration-ns-prefix "m"
   :migration-ns        'acme.migrations
   :full-schema         'acme.schema/full})

(def datomic-local      (merge datomic-base {:uri "datomic:dev://localhost:4334/acme"}))
(def datomic-staging    (merge datomic-base {:uri "datomic:dev://localhost:4334/acme-staging"}))
(def datomic-production (merge datomic-base {:uri "datomic:dev://localhost:4334/acme-production"}))
;; @c3kit/db :datomic-pro }

;; @c3kit/db :sqlite {
(def sqlite-base
  {:impl                :jdbc
   :dialect             :sqlite
   :migration-dir       "acme.migrations"
   :migration-ns-prefix "m"
   :migration-ns        'acme.migrations
   :full-schema         'acme.schema/full})

(def sqlite-local      (merge sqlite-base {:connection-uri "jdbc:sqlite:db/dev.sqlite"}))
(def sqlite-staging    (merge sqlite-base {:connection-uri "jdbc:sqlite:db/staging.sqlite"}))
(def sqlite-production (merge sqlite-base {:connection-uri "jdbc:sqlite:db/production.sqlite"}))
;; @c3kit/db :sqlite }

;; @c3kit/db :postgres {
(def postgres-base
  {:impl                :jdbc
   :dialect             :postgres
   :migration-dir       "acme.migrations"
   :migration-ns-prefix "m"
   :migration-ns        'acme.migrations
   :full-schema         'acme.schema/full})

(def postgres-local      (merge postgres-base {:connection-uri "jdbc:postgresql://localhost/acme_dev"}))
(def postgres-staging    (merge postgres-base {:connection-uri "jdbc:postgresql://localhost/acme_staging"}))
(def postgres-production (merge postgres-base {:connection-uri "jdbc:postgresql://localhost/acme_production"}))
;; @c3kit/db :postgres }

;; memory backend is always defined (HEAD-default + a valid wizard choice)
(def memory-local      {:impl :memory :full-schema 'acme.schema/full})
(def memory-staging    {:impl :memory :full-schema 'acme.schema/full})
(def memory-production {:impl :memory :full-schema 'acme.schema/full})

(def email-to-log {:client :to-log})
(def admin-email "Acme <admin@acme.com>")

(def development
  (assoc base
    :email email-to-log
    :bucket memory-local                                              ;; HEAD default; replaced by line below at scaffold
    ;; @c3kit/db :datomic-pro = :bucket datomic-local
    ;; @c3kit/db :sqlite      = :bucket sqlite-local
    ;; @c3kit/db :postgres    = :bucket postgres-local
    ;; @c3kit/db :memory      = :bucket memory-local
    :host "http://localhost:8123"
    :log-level :trace
    ;; @c3kit/feature :auth = :jwt-secret "ACME_DEV_SECRET"
    ))

;; staging and production follow the same pattern — :bucket memory-{staging|production} at HEAD
;; with the four @c3kit/db = :bucket {datomic|sqlite|postgres|memory}-{staging|production} markers below.
```

**Conflict handling at scaffold time:** the marker stripper inserts the selected `:bucket` line, then the post-scaffold hook (§7) deletes the literal HEAD-default `:bucket memory-local` line so the resulting `assoc` has exactly one `:bucket` entry. This is a hook responsibility, listed in §7 step 4b.

**Why distinct var names matter:** at HEAD, all four DB-marker blocks compile (markers are line comments). Reusing a `db-local` symbol across blocks would shadow at HEAD compilation. Distinct names (`datomic-local`, `sqlite-local`, etc.) eliminate the shadow.

```clojure
                     ["/user/reset-password/:recovery-token" :post] acme.user.api/api-reset-password
                     ["/user/social/:provider" :post]               acme.user.api/api-social-auth
                     ;; @c3kit/feature :auth }
                     })
        ;; @c3kit/feature :csp {
        csp-extra (wire.routes/lazy-routes
                    {["/v1/csp-report" :post] acme.security.csp/csp-report-handler})
        ;; @c3kit/feature :csp }
        ]
    (-> (if csp-on? (compojure/routes primary csp-extra) primary)
        ;; with :csp off, the (if csp-on? …) line + the let binding are gone via :csp markers above; result is plain `primary`
        ;; @c3kit/feature !:csp { ;; inverse — kept only when :csp off
        ;; (the let binding above is empty; we simplify the threading body
        ;; @c3kit/feature !:csp }
        (rest/wrap-rest {:keywords? true})
        (wire.routes/wrap-prefix "/api" api-not-found-handler))))
```

Similar marker pattern in `web-routes-handlers` (gates signin/social/signout/forgot/recover/websocket routes behind `:auth`) and in `defroutes handler` (gates `content-routes-handler`).

Implementation note for the plan: where marker blocks tangle threading (`let` body uses `csp-on?` only when `:csp` is on), refactor the let binding into a helper fn so the marker block is a clean single span. Will be flagged during implementation.

## 7. Post-Scaffold Hook (`c3kit-template.bb`)

Path: `templates/full-stack-reagent/c3kit-template.bb`. Invoked by the CLI after marker stripping and rename, before the atomic move (per CLI sub-spec §7.2 stage 3). The CLI passes the scaffold dir as a positional argument.

**Hook responsibilities (in order):**

1. **Read scaffold context** — read `.c3kit-create-context.edn` written by the CLI into the staging dir (contains `:name`, `:db`, selected `:features`, `:tokens` resolved variants). [CLI sub-spec note: this context file is not in §7.2; adding it requires a small CLI addition — flagged as a CLI-side prerequisite in the plan.]
2. **Generate `bin/db`** based on `:db`:
   - `:datomic-pro` → script that ensures `~/.c3kit/datomic-pro/` is present (downloads `datomic-pro-<version>.zip` and unzips if absent), runs `bin/transactor config/dev-transactor.properties`. Prompts the user on first run before downloading.
   - `:sqlite` → `mkdir -p db && touch db/dev.sqlite` + `echo "SQLite ready at db/dev.sqlite"`.
   - `:postgres` → `createdb <name>_dev || echo 'createdb failed — install Postgres and re-run'`. README explains brew/apt install.
   - `:memory` → `echo "In-memory backend; no DB process needed. Bucket starts with the app."` and exits 0.
3. **`chmod +x bin/db`**.
4. **Reconcile `:bucket` in `config.clj`** (per §6.3): delete the literal HEAD-default `:bucket memory-{local,staging,production}` lines so the marker-inserted `:bucket` value is the only one per env. The hook does a targeted line-delete keyed by an inline tag (`;; HEAD default; replaced by line below at scaffold`).

5. **Drop the seed alias** from `deps.edn` when `:auth` is off (the deps.edn marker covers this, but the hook is the explicit owner).
6. **Final sanity grep**: ensure no `@c3kit/feature` / `@c3kit/db` markers remain anywhere under the scaffold dir. Exit non-zero if any found (defensive — the CLI strips them, but the hook double-checks).
7. **Print** a one-line "DB script generated for `:<db>`" to stdout.

The hook is a bb script (`#!/usr/bin/env bb`) and has access to bb stdlib (`babashka.fs`, `babashka.process`, `clojure.edn`). No external deps.

## 8. CI Workflow

Path: `.github/workflows/template-full-stack-reagent.yml` in the c3kit-starter repo.

**Triggers:**
- Push to `main` touching `templates/full-stack-reagent/**` or `cli/**`.
- PRs touching the same.
- Manual `workflow_dispatch` for full matrix.

**Jobs:**

1. **Build CLI uberscript** — runs on `ubuntu-latest`. `cd cli && bb uberscript dist/c3kit-create.bb -m c3kit-create.main`. Uploads as artifact.

2. **Scaffold matrix** — depends on (1). Runs on `ubuntu-latest`. Matrix (Phase 1):

   ```yaml
   matrix:
     include:
       - db: memory     ;; features: defaults
       - db: sqlite     ;; features: defaults
       - db: postgres   ;; features: defaults    (uses postgres service container)
       - db: datomic-pro;; features: defaults    (downloads transactor with cache)
       - db: memory     ;; features: minimal     (--feature content=false ssr=false markdownc=false auth=false)
       - db: memory     ;; features: csp-on      (--feature csp=true)
       - db: memory     ;; features: no-auth     (--feature auth=false)
       - db: memory     ;; features: no-ssr-no-content (--feature ssr=false content=false)
   ```

   For each combo:

   ```sh
   ./c3kit-create test-app \
     --template-dir templates \
     --template full-stack-reagent \
     --db <matrix.db> \
     <matrix feature flags …> \
     --yes --no-git
   cd test-app
   bin/db &                           # background for postgres/datomic-pro
   clj -M:test:migrate
   clj -M:test:spec
   clj -M:test:cljs once
   # grep checks
   ! grep -rE '(^|[^a-z])acme($|[^a-z])' src spec dev resources content || (echo "stray acme"; exit 1)
   ! grep -rE '@c3kit/(feature|db)' . || (echo "stray marker"; exit 1)
   ```

3. **Macos smoke** — `macos-latest`, single combo (`memory + defaults`), to catch macOS-only path/permission issues. Fast.

4. **Template-as-source check** — `cd templates/full-stack-reagent && clj -M:test:spec` against the unscaffolded template tree. Verifies the template is a runnable Clojure project at HEAD.

**CI feature-flag syntax assumption:** The CLI must accept per-feature CLI flags like `--feature auth=false`. This is in the CLI sub-spec §3 implicitly (`--yes` uses defaults; per-flag override is a clean extension). Flagged in the plan as a small CLI addition required before T1 CI works.

**Datomic-Pro CI caching:** First run downloads `datomic-pro-<pinned-version>.zip` to `~/.cache/datomic-pro/`. GH Actions `actions/cache@v4` keys on the version pin. Subsequent CI runs restore from cache.

## 12. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Marker blocks in `routes.clj` / `http.clj` make the template tree hard to read/maintain | Audit during plan. Worst case: refactor by introducing per-feature ns barriers (e.g., `acme.auth.routes`, `acme.csp.middleware`) so most markers become `:delete-when-off` whole-file deletes |
| Datomic-Pro transactor download in CI is flaky | Cache + pin version. If cache miss flakes, fall back is `:memory` backend (already in matrix as default). Document version pin in the scaffolded README |
| AWS SDK removal breaks ergonomics for users deploying to AWS staging/production | Scaffolded README has explicit "Production Email" + "AWS Deployment" sections with copy-paste instructions to add SES back. Email defmethod stub left in code with a clear TODO comment |
| Postgres CI service container slows the matrix | Run postgres + datomic combos in their own jobs (parallel matrix entries). Default-on-PR matrix uses only `:memory + :sqlite` for fast turnaround; full matrix on `main` push only |
| Feature marker collides with text in `README.scaffold.md` examples (literal `@c3kit/feature` strings) | Markers in README's code blocks are escaped (use `&#64;c3kit/feature` or a sentinel like `[at]c3kit`); plan task verifies. CI grep step excludes README files from marker-residue check |
| Template tree at HEAD won't run `clj -M:test:spec` because of marker comment lines breaking forms | Markers are line-level comments only (per CLI sub-spec §6.1); they're invisible to the Clojure reader. CI step #4 (template-as-source check) verifies |
| `acme.email` ships only `:to-log`; forgot-password flow sends real-ish email in test | When `:auth` on, `acme.email/client-send-email` `:to-log` impl logs to stdout — sufficient for dev. Specs assert on `c3kit.apron.log` captured output |
| Per-env config drift (dev/staging/production) confuses single-environment OSS users | README scaffolded section explicitly notes "staging is optional; delete the `staging` def in `acme.config` if unused" |
| Hook needs a context file that the CLI doesn't write yet | Listed in §10 as a CLI prerequisite. T1 plan Task 0 ensures CLI lands this before T1 implementation lands |

## 13. Success Criteria

End of T1 implementation:

- `cd templates/full-stack-reagent && clj -M:test:spec` passes against the un-scaffolded template tree (Clojure reads through the marker comments; `acme` runs as a real project).
- CI matrix (§8) green: every `{db × feature-combo}` combo scaffolds cleanly via the CLI, runs `clj -M:test:spec`, runs `clj -M:test:cljs once`, has no residual `acme`/`Acme`/`ACME_` strings, has no `@c3kit/feature` / `@c3kit/db` markers post-scaffold.
- For at least one combo (`:memory + defaults`), `bin/db && clj -M:test:migrate && clj -M:test:seed && clj -M:test:run` boots the server and serves `GET /` 200 in CI.
- Scaffolded `README.md` correctly describes the user's chosen backend + features.
- `acme.email :ses` impl is fully removed; AWS SDK deps absent from `deps.edn`.
- `my.datomic.com` mvn-repos entry removed; `com.datomic/peer` resolves from Maven Central.
- Marker block count ≤30 inline blocks; auth feature ≤14 blocks (refactor if not).
- `LICENSE` (MIT) present at template root.
