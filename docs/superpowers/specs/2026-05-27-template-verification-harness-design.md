# Template Verification Harness — Design

Date: 2026-05-27
Status: Approved (design)
Scope: **Harness only** — build the verification method. It surfaces defects as failing
checks; it does **not** fix the underlying template/CLI bugs. Fixes are a later pass.

## Problem

Projects scaffolded from `templates/full-stack-reagent` ship defects that no automated
check currently catches:

1. **Cruft copied into the scaffold.**
   - `render-into-stage!` (`cli/src/c3kit_jig/main.clj`) does `fs/copy-tree` of the
     template dir **from disk**, so untracked/ignored artifacts are copied verbatim:
     `full-stack-reagent.iml`, `.cpcache/` (and would include `target/`, `node_modules/`,
     etc. if present).
   - `render!` (`cli/src/c3kit_jig/render.clj`) deletes only a hardcoded short list
     (`c3kit-template.edn`, `c3kit-template.bb`, `spec/hook_test.bb`,
     `.c3kit-jig-context.edn`). Template-dev tooling that lives *inside* the template —
     `dev/verify-scaffold.bb` and `spec/combos/` — is not deleted and ships in the scaffold.
     Root cause: verification tooling is colocated with the template payload.

2. **Namespace bug.** `ns` forms render with an underscore project prefix
   (`(ns probe_app.core …)`) instead of the hyphenated form (`(ns probe-app.core …)`).
   Root cause: the template token `acme` is a single word, so in
   `cli/src/c3kit_jig/rename.clj/replace-token` the `:underscore` and `:hyphen` source
   spellings are identical (`"acme"`); the `cond->` applies `:underscore` before
   `:hyphen`, consuming every match. A single-word token cannot distinguish ns-symbol
   context (wants hyphen) from path-string context such as `resources/config/cljs.edn`
   `:ns-prefix` (wants underscore) by string matching alone.

3. **Noisy clj specs.** `clojure -M:test:spec` passes but prints log lines (timestamps /
   level markers) mixed into the test output.

4. **cljs specs do not run non-interactively.** The `:cljs` alias runs
   `acme.compile-cljs`, which calls the c3kit scaffold cljs runner and ends in
   auto-watch (`c3kit.scaffold.cljs` `"auto"` mode) — it never exits, so it is unusable
   in CI. A one-shot mode exists: `c3kit.scaffold.cljs` `"once"` (build + run specs once,
   then exit), driven through Playwright headless browser.

### Non-issues (verified, not in scope)

- **SSR feature gating works.** A fresh scaffold with `ssr=false` correctly removes all
  `*/ssr/` dirs, `package.json`, and `resources/prerender*`. The SSR leftovers seen in the
  `cli/my-cool-app` / `cli/my-test-app` samples are generated by `acme.compile-cljs`
  *after* scaffold (`resources/prerender`, `dev/<ns>/ssr/prerender_pages.cljs`), and those
  samples are stale. A latent issue — `compile_cljs.clj` runs SSR-specific code even when
  SSR is off — is noted but **out of scope** (fix pass).
- **Specs need no live database.** Every clj spec runs against `{:impl :memory}`
  (`acme.test-data/with-memory-kinds` and `with-memory-schema`). No spec targets a real
  backend; the only datomic reference is `datomic:mem://test` (in-process, unused). The
  chosen DB backend only affects `deps.edn` and `acme.config`, not spec execution. Running
  clj specs per-combo therefore requires no DB services — it verifies that backend's
  `deps.edn` resolves and `config.clj` loads; the assertions run on memory either way.

## Goals

A reusable harness that, for a scaffolded project, verifies:
- no forbidden artifacts/cruft are present,
- namespaces are rendered correctly (hyphenated ns forms),
- clj specs pass **and** produce clean output (no stray log lines),
- cljs specs actually run and pass (one-shot),
- the dev server boots.

The harness must **not live inside any template** (so it cannot leak into scaffolds), and
must be **template-agnostic within the Clojure/c3kit family**: the engine and checks are
shared; each template declares, as data, which checks/commands/combos apply. It is **not**
required to support non-Clojure templates or non-c3kit Clojure stacks. Templates may vary
(e.g. clj-specs-only, cljs-specs-only); the descriptor toggles checks accordingly.

## Architecture (Approach A — extend & relocate the existing bb harness)

Reuse the proven scaffold-and-assert engine of today's `verify-scaffold.bb`, but relocate
it out of the template and generalize per-template specifics into a declarative descriptor.

```
verification/
  bb.edn                          # tasks: verify (one combo), verify-all
  engine.bb                       # entry point; no per-template knowledge
  checks.bb                       # the c3kit check set (enabled/disabled by descriptor)
  templates/
    full-stack-reagent/
      verify.edn                  # per-template descriptor (see below)
      combos/
        memory-defaults.expected.edn
        memory-minimal.expected.edn
        memory-no-auth.expected.edn
        memory-no-ssr-no-content.expected.edn
        memory-no-websocket.expected.edn
        sqlite-defaults.expected.edn
        postgres-defaults.expected.edn
        datomic-pro-defaults.expected.edn
```

`combos/` is **moved** from `templates/full-stack-reagent/spec/combos/`, and
`dev/verify-scaffold.bb` is **deleted** from the template. This relocation removes those two
items from the scaffold leak at the source, without touching the CLI copy mechanism (which
stays in the fix pass).

### Engine

`engine.bb`:
- Args: `--template <id>` (default `full-stack-reagent`), `--combo <name>`,
  `--tier full|light`, `--cli-cp <dir>` (default `../cli/src`), `--keep-tmp`.
- Loads `verification/templates/<id>/verify.edn`.
- Scaffolds the combo into a temp dir by invoking the CLI exactly as today:
  `bb -cp <cli-cp> -m c3kit-jig.main create <name> --template-dir <templates-dir>
  --template <id> --db <db> --yes --no-git <feature flags…>`.
- Runs the checks enabled for the tier (intersection of descriptor `:checks` and the tier's
  default set), each returning `{:check <kw> :ok? <bool> :detail <str>}`.
- Prints one line per check; exits non-zero if any check fails.
- Cleans the temp dir unless `--keep-tmp`.

### Descriptor (`verify.edn`)

Declarative, per template. Example for full-stack-reagent:

```clojure
{:combos {:memory-defaults          {:tier :full}
          :memory-minimal           {:tier :light}
          :memory-no-auth           {:tier :light}
          :memory-no-ssr-no-content {:tier :light}
          :memory-no-websocket      {:tier :light}
          :sqlite-defaults          {:tier :light}
          :postgres-defaults        {:tier :light}
          :datomic-pro-defaults     {:tier :light}}

 :denylist        ["dev/verify-scaffold.bb" "spec/combos"]   ; extras beyond .gitignore + standard template-dev files
 :ns-prefix-exempt ["resources/config/cljs.edn"]             ; underscore prefix is correct here

 :commands {:clj-spec  ["clojure" "-M:test:spec"]
            :cljs-once  ["clojure" "-M:test:cljs" "once"]
            :migrate    ["clojure" "-M:test:migrate"]
            :run        ["clojure" "-M:test:run"]
            :port       8123}

 :checks  {:no-cruft true :combo true :residue true :ns-hyphen true
           :clj-clean true :cljs-run true :server-boot true}}
```

A clj-only template would set `:cljs-run false`; a cljs-only template would set
`:clj-clean false`/`:server-boot false`. Checks are the known c3kit set toggled here — not
an open plugin registry.

### Checks (`checks.bb`)

| Check | Tier | Rule |
|---|---|---|
| `:no-cruft` | both | Scaffold contains none of: the template `.gitignore` patterns (`*.iml`, `.cpcache/`, `/target`, `node_modules/`, `resources/prerender*/`, `dev/**/ssr/prerender_pages.cljs`, …) **plus** standard template-dev files (`c3kit-template.edn`, `c3kit-template.bb`, `spec/hook_test.bb`, `.c3kit-jig-context.edn`) **plus** descriptor `:denylist` extras. |
| `:combo` | both | From the combo edn: `must-exist`, `must-not-exist`, `file-contains`, `file-not-contains` (ported verbatim from today's `verify-scaffold.bb`). |
| `:residue` | both | No `@c3kit/feature` or `@c3kit/db` markers remain in the scaffold (grep). The harness/combo files live outside the scaffold, so no self-exclusion is needed. |
| `:ns-hyphen` | both | Every `(ns <prefix>…)` form in `*.clj`/`*.cljc`/`*.cljs` uses the hyphenated project name (e.g. `probe-app.*`), never the underscore form (`probe_app.*`). Files in `:ns-prefix-exempt` are skipped. |
| `:clj-clean` | both | Run `:clj-spec` command. Pass iff exit 0 **and** no log-level lines — lines containing a timestamp or an `INFO`/`DEBUG`/`WARN`/`ERROR` marker — appear outside speclj's own progress/summary output. (speclj dots/`documentation` output + final summary are allowed.) |
| `:cljs-run` | full | Run `:cljs-once` command (Playwright headless). Pass iff exit 0, specs ran (count > 0), and 0 failures. |
| `:server-boot` | full | Run `:migrate`; start `:run` in the background with `PORT=<:port>`; poll `http://localhost:<:port>/` until any HTTP response or a timeout; then kill the process. Pass iff a response arrives before timeout. |

Tier default check sets:
- **full**: all seven.
- **light**: `:no-cruft :combo :residue :ns-hyphen :clj-clean`.

The effective set for a run = tier default set ∩ descriptor `:checks` that are `true`.

## Coverage (full-stack-reagent)

- **memory-defaults** → `:full` (all seven checks, incl. cljs + server boot).
- **all other combos** → `:light` (no cljs/server). Cheap: no DB services required because
  specs run on the memory bucket; per-combo value is confirming that backend's `deps.edn`
  resolves and `config.clj` loads.

## Invocation

- Local: `cd verification && bb verify --combo memory-defaults --tier full`.
  `bb verify-all` runs every combo at its declared tier.
- CI (`.github/workflows/template-full-stack-reagent.yml`): rewrite the `scaffold-matrix`
  job to call `verification/engine.bb` per combo at its declared tier. The memory-defaults
  (full) matrix entry adds a Playwright browser install step (for `:cljs-run`). Other
  combos run light. Remove references to the in-template `dev/verify-scaffold.bb`.
  - The existing `template-at-head` job (runs `clojure -M:test:spec` inside the template
    tree) and `hook-iso-test` job (runs `spec/hook_test.bb` inside the template) are
    unaffected: `hook_test.bb` stays in the template for that job and is already stripped
    from scaffolds by `render!`.

## Expected first-run result (scope guard)

On first run the harness is expected to be **red** for full-stack-reagent:
- `:no-cruft` — fails on `full-stack-reagent.iml` and `.cpcache/`.
- `:ns-hyphen` — fails on underscore ns prefixes.
- `:clj-clean` — fails on log noise.
- `:cljs-run` — fails (auto-watch / Playwright not wired for one-shot).

These are the defects the harness is meant to document. Fixing them is a separate,
later pass and is **not** part of this work.

## Out of scope (recorded for the fix pass)

- Hardening the CLI copy step to exclude untracked/ignored files.
- The namespace-rendering fix (context-aware hyphen vs underscore).
- Quieting clj spec log output.
- A clean one-shot cljs spec command in the template.
- Gating `compile_cljs.clj`'s SSR-specific code behind the `:ssr` feature.
